# Cycle 04 UAT: Surrogate Labeling and Pacemaker Base Training

## 1. Test Scenarios

### Scenario 01: Successful Batch Labeling
-   **Priority**: High
-   **Description**: Verify that the system can use the MACE Oracle to label a large batch of structures efficiently.
-   **Execution**:
    1.  Create `tests/uat/configs/cycle04_labeling.yaml` specifying `step5_surrogate_labeling.batch_size: 100`.
    2.  Provide a `surrogate_candidates.xyz` file (generated from Step 4).
    3.  Run the pipeline for Step 5.
    4.  Check output for "Labeled 100 structures" and verify `surrogate_dataset.pckl.gzip` is created.
    5.  Load the dataset and verify that energies and forces are non-zero.

### Scenario 02: Base ACE Training
-   **Priority**: Critical
-   **Description**: Verify that the system can train a base ACE potential from the surrogate dataset.
-   **Execution**:
    1.  Provide a `surrogate_dataset.pckl.gzip`.
    2.  Run the pipeline for Step 6 (Pacemaker Training).
    3.  Check output for "Training complete" and verify `potential.yace` is created.
    4.  Verify that the log file shows a decreasing loss (mocked).

## 2. Behavior Definitions (Gherkin)

### Feature: Surrogate Dataset Creation and ACE Training

**Scenario: Label structures with MACE**
  GIVEN a list of unlabeled structures generated by MD
  AND a fine-tuned `MaceSurrogateOracle`
  WHEN the Orchestrator executes Step 5
  THEN it should run batch prediction on the structures
  AND it should save the labeled structures to `surrogate_dataset.pckl.gzip`
  AND the dataset should contain valid energy and force values

**Scenario: Train base ACE potential**
  GIVEN a `surrogate_dataset.pckl.gzip`
  AND a `PacemakerTrainer` configuration
  WHEN the Orchestrator executes Step 6
  THEN it should generate a valid `input.yaml` for Pacemaker
  AND it should invoke the training process
  AND it should produce a `potential.yace` file
  AND the potential file should not be empty
